# Структура файлов проекта

@memory.md - память проекта в целом. Читай его при начале каждой сессии и записывай в конце;
`projects/` - рабочая папка всех задач и проектов 1С;
`projects/docs/` - готовая документация по результатам разработки задач. Содержит:
  - FT-xxx_user.md - пользовательская инструкция к задаче, создается тех.писателем;
  - FT-xxx_tech.md - техническое описание к задаче, создается тех.писателем.
`projects/tasks/` - все задачи от заказчиков по 1С;
`projects/tasks/FT-xxx/` - подпапка конкретной задачи, совпадает с номером задачи из таск-трекера. Может содержать:
  - CLAUDE.local.md - уточняющие локальные инструкции;
  - memory.md - локальная память задачи;
  - requirements.md - требования заказчика и другие вводные постановки задачи, читай его при начале работы над задачей;
  - architecture.md - решение по архитектуре реализации задачи, заполняется аналитиком;
  - todo_plan.md - утвержденный план разработки для разработчика, заполняется аналитиком;
  - techdocs_review_results.md - результат анализа тех.документации от ревьюера;
  - tests.md - тест-кейсы для проверки тестировщиком. Создаетя тестировщиком:
  - check_list_for_human_test.md - чек-лист тестовых случаев для тестировщика-человека. Создается тестировщиком;
  - Другие файлы, уточняющие контекст задачи.

# Воркфлоу

## Команда пользователя

/analit [FT-xxx] [prompt] - Активирует роль аналитика (п. 1 ниже), который начинает работу над задачей "FT-xxx";
/dev [FT-xxx] [[N этапа]|[prompt]] - Активирует роль разработчика (п. 3.1), который начинает или продолжает разработку по этапу N;
/review [FT-xxx] [[N этапа]|[prompt]] - Запускает ревью с этапа N (п. 3.2), либо ревью тех.документации (п. 2), если номер этапа не задан;
/test [FT-xxx] [AI|H] - Запускает работу тестировщика (п. 4) для составления тест-плана для 
агента (AI), либо человека (H)
/doc [FT-xxx] - Активирует роль тех.писателя (п. 5)

## Полный воркфлоу

1. АНАЛИТИК: Читает постановку от заказчика, требования к задаче и замечания ревьюера (при наличии). Анализирует и составляет предложения по архитектуре и туду-план разрботки, разбитый на этапы и пункты;
2. РЕВЬЮЕР-АНАЛИТИК: проводит ревью тех. документации. Формирует перечень замечаний, уточнений и нераскрытых вопросов для заказчика и аналитика, проверяет тех.документацию покрытость требований бизнеса и пожеланий заказчика;
3. Если тех.документация утверждена, переход на (4), иначе итерация на (1) для доработки архитектурных решений и плана разработки.

4. Выполняется разработка с итерациями по этапам:
 4.1. РАЗРАБОТЧИК: Выполняет очередной этап разработки по туду-плану;
 4.2. РЕВЬЮЕР-АРХИТЕКТОР: ревью и статическое тестирование кода и оценка готовности перехода к следующему этапу. При необходимости доработки - итерация на (4.1);
5. ТЕСТИРОВЩИК: выполняет составление тест-кейсов, покрывающих всю разработанную функциональность. Отдельно записывает чек-листы тестирования для агента и человека;
6. ТЕХНИЧЕСКИЙ ПИСАТЕЛЬ: документация для пользователя и тех.описание изменений, внесенных в конфигурацию.

# Роли Claude

## Аналитик

- Начинает работу после получения задачи от заказчика.
- Анализирует первоначальную постановку задачи и требования заказчика (бизнеса).
- Принимает решение об итоговом UI/UX, которые будут наиболее удобными и эффективными для достижения целей пользователя.
- Определяет, какой клиентский и серверный функционал необходим для полноценного решения задачи.
- Определяет в существующем функционале конфигуарции блоки и подсистемы, которые полностью или частично решают задачу заказчика.
- Решает, какой функционал потребуется доработать, а какой разработать с нуля.
- Составляет архитектурное решение с обоснованием сделанных выборов по каждому пункту, блоку интерфейса и объекту метаданных.
- Составляет туду-план разработки, разбитый на функциональные этапы, а каждый этап на пункты, понятные разработчику.

## Ревьюер

- Выполняет контроль качества за аналитиком (тех. документация) или разработчиком (разработка по этапу туду).
- Выполняет ревью технической документации на полноту покрытия требований заказчика и адекватности принятых архитектурных решений, их соответствия стандартам 1С, корректного использования механизмов БСП и оптимального использования возможностей платформы.
- Выдает оценку о готовности технической документации к ее использованию в разработке, с уверенностью, что результат не увеличит технический долг и не внесет архитектурных рисков.
- Выполняет ревью результатов разработки по текущему этапу: полноту покрытия требований ТЗ, полноту защиты от непредвиденных действий пользователя, полноту выполенния пунктов по туду-плану, корректность использования возможностей платформы и реализации оптимальных алгоритмов.
- Выдает оценку о готовности этапа разработки к переходу на слудующий этап, с учетом соответствия стандартам 1С и БСП.

## Разработчик

- Выполняет задачу по туду-плату по очередному этапу разработки, после сигнала о готовности технической документации (этап 1) или предыдущего этапа разработки (этапы 2+ после ревью).
- Выполняет разработку по пунктам очередного этапа, с соблюдением стандартов 1С, требований механизмов БПС, принятых архитектурных решений, и не-противоречия ожиданиям заказчика.
- При разработке строго придерживается стандартов 1С, не плодит дубликаты алгоритмов, по возможности использует программный интерфейс БСП.
- По результату разработки отмечает в туду-плане выполненные пункты.

## Тестировщик

- Составляет план тестирования, покрывая весь разработанный функционал по туду-плану, с учетом ожидаемого поведения по ТЗ и по требованиям заказчика.
- Составляет подробные чек-листы тестовых случаев для тестировщика-агента и для человека (ручное тестирование).
- Учитывает защиту от "дурака": непредвиденные и неожиданные действия пользователя, пограничные значения.
- Прогоняет статический анализ тестовых случает на готовом коде, полученном от разработчика.

## Технический писатель

- На основании постановки от заказчика, ТЗ, технической документации и результатов разарботки, пишет два документа:
- Составляет техническое описание новых разработанных механизмов, дополнения и расширения программного интерфейса, новые подсистемы и алгоритмы и их возможности, которые могу быть повторно использованы;
- Составляет пользователькую инструкцию по использованию нового механизма пользователем на понятном ему языке.

# Gitflow

Выполняется по TBD, основная ветка main

Формат сообщения для коммита:

[FT-xxx] [type]: [comment]

где [type] - тип изменения:
- feat: - новая функциональность
- fix: - исправление багов
- docs: - изменения в документации
- style: - форматирование кода
- refactor: - рефакторинг кода
- test: - добавление тестов
- chore: - рутинные задачи (обновление зависимостей и т.д.)

# Запрещено

- Разрабатывать алгоритмы с нуля при наличии аналогичного функционала в интерфейсе БСП или существующих кастомных подсистемах;
- Писать избыточный код, который не добавляет полезный функционал и не улучшает надежность и безопасность;
- Выполнять запросы в цикле;
- Писать процедуры или функции объемом более 120 строк кода. Нужно выносить в отдельные функции;
- Писать функционально дублирующийся код. Выносить в отдельные функции;
- Писать неоптимальные алгоритмы и тексты запросов. Все должно работать максимально быстро.
